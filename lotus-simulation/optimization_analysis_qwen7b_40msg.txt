============================================================
C) Why zep still "looks less vLLM-driven" than mem0/evermemos
============================================================
1) zep insertion is dominated by sem_join operations.
2) LOTUS sem_join optimizer currently centers on similarity-proxy path; helper-lm is not the primary join-proxy execution path.
3) Therefore, even with helper_lm configured, zep's practical acceleration depends more on join-cascade behavior + cache + fallback rate than on direct helper-lm routing.

============================================================
E) Root-Cause Attribution
============================================================
Workflow-driven factors
- mem0 and evermemos each have sem_filter choke points where helper cascade cleanly applies.
- zep's core workload is sem_join-heavy and is more sensitive to join optimizer internals.

LOTUS-implementation factors
- sem_join helper-lm proxy path is not the dominant join optimization mechanism in the current code path.
- join cascade can still work via similarity-proxy planning, but may incur instability in specific edge-cases.

Script/data interaction factors
- zep Q5 repeatedly triggers index out-of-bounds fallback under current data trajectory, reducing effective optimization coverage.
- evermemos runtime is strongly impacted by boundary/segmentation outcomes, not only per-call cascade routing.
